#+TITLE: Elm Architecture Refactor (TUI/GUI Shared UI Loop)
#+DATE: 2026-03-31

* Motivation

The current TUI code mixes concerns: ~main.rs~ maps events to actions, applies
state changes inline, and dispatches side effects all in one place. ~app.rs~
combines state, input handling, and rendering. This makes it hard to reuse the
UI logic when we build a GUI — every state transition would need to be
reimplemented.

The Elm architecture (Model-View-Update), which iced adopts, cleanly separates:

- *Model*: the entire application state, plain data.
- *Message*: a description of something that happened (user input, agent
  response, timer tick). Messages are the /only/ way to change the model.
- *update()*: a pure function ~(Model, Message) -> (Model, Vec<Command>)~ that
  computes the next state and any side effects to perform.
- *view()*: a pure function ~(Model) -> UI~ that renders the model. In ratatui
  this is ~fn view(frame, &Model)~; in iced it returns an ~Element~. The
  rendering backend changes, but the model doesn't.
- *Command*: a description of a side effect (send message to agent, quit, etc).
  The runtime executes commands and feeds their results back as new messages.

By adopting this pattern now, the future GUI can reuse the *Model*, *Message*,
*update()*, and *Command* types wholesale — only *view()* and the runtime loop
change.

* File layout after refactor

| File          | Contains                                                     |
|---------------+--------------------------------------------------------------|
| ~model.rs~    | ~Model~ struct (all application state), ~ChatMessage~, etc   |
| ~message.rs~  | ~Message~ enum (every possible state change)                 |
| ~update.rs~   | ~fn update(model, msg) -> Vec<Command>~ (pure state logic)   |
| ~command.rs~  | ~Command~ enum (side effects the runtime must execute)       |
| ~view.rs~     | ~fn view(frame, &Model)~ + all render_* helpers (ratatui)    |
| ~event.rs~    | ~EventHandler~ (unchanged, maps crossterm -> Message)        |
| ~theme.rs~    | ~Theme~ struct (unchanged)                                   |
| ~terminal.rs~ | Terminal init/restore helpers (ratatui-specific)             |
| ~lib.rs~      | Re-exports                                                   |

~app.rs~ is removed — its contents are split across the above files.

** DONE Step 4: Implement ~update()~ in ~update.rs~

Pure function: takes ~&mut Model~ and ~Message~, returns ~Vec<Command>~. All
state transitions live here. No I/O, no channels, no rendering.

Key design rules (matching iced/Elm):
- Every ~Message~ variant has exactly one match arm.
- State changes are immediate mutations on ~Model~.
- Side effects are returned as ~Command~ values, never executed inline.
- The function is testable without a terminal or network.

** DONE Step 5: Move rendering to ~view.rs~

Move ~render()~, ~render_status_bar()~, ~render_output()~, ~render_input()~,
~render_debug_panel()~, ~render_palette()~, and ~palette_commands()~ from
~app.rs~ into ~view.rs~.

The only signature change: ~fn view(frame: &mut Frame, model: &Model)~.

This module is the *only* ratatui-specific code. When building a GUI, this file
gets replaced with an iced ~view()~ function — everything else stays.

** DONE Step 6: Add event-to-message mapping

A small function that converts crossterm ~Event~ into ~Message~:

#+begin_src rust
pub fn map_event(event: &event::Event, model: &Model) -> Option<Message> { ... }
#+end_src

This lives in ~event.rs~ or a new ~subscription.rs~. It reads model state (e.g.
~palette_open~, ~is_streaming~, ~debug_mode~) to decide which message to
produce. The function replaces the current ~handle_key()~ function.

** DONE Step 7: Refactor ~main.rs~ to the MVU runtime loop

The main loop becomes:

#+begin_src rust
loop {
    // View
    terminal.draw(|frame| view::view(frame, &model))?;

    // Wait for next event
    tokio::select! {
        event = events.next() => {
            if let Some(msg) = map_event(&event?, &model) {
                let cmds = update::update(&mut model, msg);
                for cmd in cmds {
                    execute_command(cmd, &req_tx);
                }
            }
        }
        Some(agent_event) = agent_rx.recv() => {
            let msg = map_agent_event(agent_event);
            let cmds = update::update(&mut model, msg);
            for cmd in cmds {
                execute_command(cmd, &req_tx);
            }
        }
    }

    if model.should_quit { break; }
}
#+end_src

~execute_command()~ is the only place with side effects — it sends messages
over channels or sets quit flags. It's the runtime's job, not the model's.

* What stays the same for the GUI

When we build the iced GUI, these modules are *reused as-is*:
- ~model.rs~ — same state struct
- ~message.rs~ — same message enum
- ~update.rs~ — same update logic
- ~command.rs~ — same command enum
- ~theme.rs~ — colors adapt to iced styles

These modules get *replaced*:
- ~view.rs~ — ratatui rendering replaced with iced ~Element~ tree
- ~event.rs~ — crossterm events replaced with iced subscriptions
- ~terminal.rs~ — not needed in GUI

* Notes

- The ~Command~ enum will grow over time (e.g. ~SwitchProvider~, ~Undo~,
  ~ForkSession~). Each new command adds a variant and a handler in the runtime.
- The ~Message~ enum mirrors iced's approach where every widget interaction
  produces a message. This is verbose but explicit — no hidden state changes.
- ~update()~ being a free function (not a method on Model) makes it easy to
  test: construct a Model, send a Message, assert the resulting state.
