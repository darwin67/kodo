* Motivation

Auth currently requires manually setting environment variables
(~ANTHROPIC_API_KEY~, ~OPENAI_API_KEY~, etc.). This is friction for new users
and doesn't support token refresh. This plan removes the env var requirement
entirely — credentials are stored in the OS keychain and obtained via the
~--login~ flow.

* Scope

Implement per-provider auth strategies:
- *OpenAI*: OAuth2 Authorization Code + PKCE (matching the flow Codex CLI
  uses), but store ChatGPT OAuth tokens/account metadata directly rather than
  assuming the login can always be converted into an API-key-style credential.
- *Anthropic*: API key only (keychain -> prompt via ~--login~). Anthropic does
  not expose a public OAuth flow for direct API access.
- *Gemini*: API key via ~--login gemini~ prompt, stored in keychain.

A unified ~resolve_auth()~ chain handles: keychain (with refresh if supported)
-> error with instructions to run ~--login~.

As part of verification, the in-app slash command flow must also support:
- Listing the coding-capable models available to the current credentials.
- Selecting one of those models so the user can immediately test simple prompts
  after login.

* Provider Auth Matrix

| Provider  | Auth Method             | OAuth Support | Token Refresh          | Header Format                                 |
|-----------+-------------------------+---------------+------------------------+-----------------------------------------------|
| OpenAI    | ChatGPT OAuth (PKCE)    | Yes           | Yes                    | ~Authorization: Bearer <token>~ + ~ChatGPT-Account-ID~ |

* Reference: Codex CLI OAuth Flow

The Codex CLI (https://github.com/openai/codex) implements OAuth2 Authorization
Code + PKCE against OpenAI's auth infrastructure. We replicate this flow for
our OpenAI provider.

** Key Constants

- *Issuer*: ~https://auth.openai.com~
- *Client ID*: ~app_EMoamEEZ73f0CkXaXp7hrann~ (public client, no secret)
- *Authorize URL*: ~https://auth.openai.com/oauth/authorize~
- *Token URL*: ~https://auth.openai.com/oauth/token~
- *Redirect URI*: ~http://localhost:1455/auth/callback~
- *Scopes*: ~openid profile email offline_access api.connectors.read api.connectors.invoke~
- *Extra authorize params*: ~id_token_add_organizations=true~,
  ~codex_cli_simplified_flow=true~, ~originator=codex_cli_rs~
- *PKCE*: S256, 64-byte random verifier, base64url-encoded (no padding)

** Authorization Code Flow (primary)

1. Start local HTTP server on ~localhost:1455~.
2. Build authorize URL with ~response_type=code~, ~client_id~, ~redirect_uri~,
   ~scope~, ~code_challenge~ (S256), ~code_challenge_method=S256~, ~state~,
   ~id_token_add_organizations~, ~codex_cli_simplified_flow~, and
   ~originator~.
3. Open user's browser to authorize URL.
4. Receive callback at ~/auth/callback~ with ~code~ and ~state~ params.
5. Exchange code for tokens via POST to token URL:
   #+begin_src text
   grant_type=authorization_code
   code={authorization_code}
   redirect_uri={redirect_uri}
   client_id={client_id}
   code_verifier={pkce_verifier}
   #+end_src
6. Response contains: ~id_token~, ~access_token~, ~refresh_token~.

** Investigation Findings (2026-04-15)

- Browser OAuth is now confirmed working end-to-end through the local callback:
  the browser reaches ~http://localhost:1455/auth/callback~ and shows
  "Login successful! You can close this window."
- The fixed callback port, extra authorize params, expanded scopes, and
  explicit connection close on the callback response were all required to make
  the browser flow reliable.
- Kodo's TUI also needed explicit in-app notices during login. Printing the
  login banner to stderr while the TUI was active was not sufficient.
- The authorization-code exchange succeeds and returns ChatGPT OAuth tokens.
- The follow-up RFC 8693 token exchange for ~requested_token=openai-api-key~
  is *not* a reliable prerequisite for login in this environment. Live testing
  hit:
  - ~401 Unauthorized~ with ~"Invalid ID token: missing organization_id"~
  - ~400 Bad Request~ with ~"Unknown parameter: 'organization_id'"~ when
    attempting to supply that parameter manually
- Inspecting the current Codex implementation shows that Codex does *not*
  depend on the API-key exchange to function. It stores ChatGPT OAuth tokens as
  first-class auth, uses the OAuth ~access_token~ as the bearer token, and also
  sends ~ChatGPT-Account-ID~ derived from the ~id_token~.
- For verification parity, successful login is not enough by itself. Kodo
  should also be able to enumerate coding-capable models for the active auth
  and select one in-app so a trivial prompt can confirm the credentials work
  end-to-end.
- Therefore the original plan assumption "OpenAI login returns an API key and
  Kodo stores that as the provider secret" is incorrect for parity with current
  Codex. The OpenAI provider path must support a ChatGPT-auth mode directly.

** Optional API Key Exchange (post-login)

After obtaining tokens, Codex performs a second token exchange (RFC 8693) to
get an actual OpenAI API key:

#+begin_src text
POST https://auth.openai.com/oauth/token
grant_type=urn:ietf:params:oauth:grant-type:token-exchange
client_id={client_id}
requested_token=openai-api-key
subject_token={id_token}
subject_token_type=urn:ietf:params:oauth:token-type:id_token
#+end_src

Response: ~{ access_token }~ — this is the API key used for subsequent requests.

*Important finding*: In the current Codex implementation this step is
best-effort and is *not* the core runtime auth path. Codex persists ChatGPT
OAuth credentials even if API-key exchange is unavailable and continues by
using the OAuth ~access_token~ plus ~ChatGPT-Account-ID~.

** Token Refresh

#+begin_src text
POST https://auth.openai.com/oauth/token
Content-Type: application/json
{
  "client_id": "app_EMoamEEZ73f0CkXaXp7hrann",
  "grant_type": "refresh_token",
  "refresh_token": "{refresh_token}"
}
#+end_src

Codex proactively refreshes when the ChatGPT OAuth access token is within
8 minutes of expiry.

** Device Code Flow (headless fallback)

For SSH/headless environments where a browser can't be opened locally:

1. POST to ~{issuer}/api/accounts/deviceauth/usercode~ with ~{ client_id }~
   → returns ~{ device_auth_id, user_code, interval }~.
2. Display: "Go to https://auth.openai.com/codex/device and enter code: {user_code}".
3. Poll ~{issuer}/api/accounts/deviceauth/token~ with ~{ device_auth_id, user_code }~
   every ~interval~ seconds (15 min timeout).
4. On success: returns ~{ authorization_code, code_challenge, code_verifier }~.
5. Exchange code using the same token endpoint.

* Reference: Gemini API Auth

- HTTP header: ~x-goog-api-key: {key}~.
- Obtain at https://aistudio.google.com/app/apikey.
- No expiry, no refresh needed.
- Sufficient for all generation endpoints.

** DONE Step 4: Add ~refresh_if_needed()~ to ~auth.rs~

*What*: Check token expiry and refresh if needed. Only applies to OAuth tokens
(OpenAI); API keys (Anthropic, Gemini) never expire.

*File*: ~crates/kodo-store/src/auth.rs~

~refresh_if_needed(pool, store, account_id, token_url, client_id)~:
1. Load account from keychain; return ~None~ if not found.
2. If no ~expires_at~: return token as-is (API key, never expires).
3. If expires more than 8 minutes from now: return token as-is.
4. If expired and no ~refresh_token~: return ~None~.
5. POST refresh to ~token_url~:
   - OpenAI: JSON body ~{ client_id, grant_type: "refresh_token", refresh_token }~
   - Standard: form-encoded same fields
6. On success: persist refreshed ChatGPT OAuth ~access_token~, ~refresh_token~,
   and parsed account metadata.
7. Compute new ~expires_at~ from ~expires_in~; call ~update_token()~; return the
   refreshed auth bundle.
