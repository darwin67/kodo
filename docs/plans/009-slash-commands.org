* Motivation

Provider and model management currently has no in-app interface — users must
restart with CLI flags. A slash command system (like Claude Code's ~/model~,
~/clear~, etc.) lets users manage accounts, discover credential-usable coding
models, switch models, and control the session without leaving the input
prompt.

* Scope

When the user types ~/~ at the start of the input buffer, kodo enters slash
command mode: an autocomplete list appears above the input, narrows as the user
types, and the selected command executes on Enter. Escape or backspace-to-empty
cancels back to normal input.

Built-in commands:

| Command                    | Effect                                      |
|----------------------------+---------------------------------------------|
| ~/help~                    | Print all available commands                |
| ~/clear~                   | Clear conversation history                  |
| ~/compact~                 | Summarise context and replace messages      |
| ~/model [id]~              | List current credential-usable models, show current selection, or switch to ~id~ |
| ~/providers~               | List all connected accounts                 |
| ~/login <provider> [name]~ | Add a new account (OAuth or API key prompt) |
| ~/logout <account_id>~     | Remove an account from keychain + DB        |

Credential verification requirement:
- After login, users should be able to use slash commands to list the models
  available for coding with the current credentials and select one for simple
  prompt tests.

* Current State

- Input is a plain text buffer; no special handling of leading ~/~.
- ~command.rs~ has a ~PaletteCommand~ list used for the ~Ctrl+P~ palette.
- No slash command parser, no inline autocomplete.

* Skill Customisation

Users can define custom slash commands by placing a ~SKILL.md~ file (following
the [[https://agentskills.io][Agent Skills]] open standard) inside a ~skills/<name>/~ directory.  Kodo
discovers skills at two scopes:

| Scope    | Path                                   | Applies to         |
|----------+----------------------------------------+--------------------|
| Personal | ~~/.kodo/skills/<name>/SKILL.md~       | All projects       |
| Project  | ~.kodo/skills/<name>/SKILL.md~         | This project only  |

Project skills take precedence over personal ones when names collide.  Kodo
also accepts the legacy flat layout ~.kodo/commands/<name>.md~ for backward
compatibility; a same-name skill directory wins over a commands file.

** DONE Step 11: ~SkillDef~ type and ~SKILL.md~ parser

*What*: Parse a ~SKILL.md~ file into a typed ~SkillDef~ struct.  The file
format is a YAML frontmatter block (between ~---~ markers) followed by
free-form markdown body content.

*File*: ~crates/kodo-ui/src/skills.rs~ (new file)

Frontmatter fields to support:

| Field                      | Type             | Default    |
|----------------------------+------------------+------------|
| ~name~                     | ~Option<String>~ | dir name   |
| ~description~              | ~Option<String>~ | first para |
| ~argument-hint~            | ~Option<String>~ | none       |
| ~disable-model-invocation~ | ~bool~           | ~false~    |
| ~user-invocable~           | ~bool~           | ~true~     |

Structs / functions:

- ~SkillFrontmatter~: serde-deserialised from the YAML block (all fields
  optional).
- ~SkillDef~: resolved struct with ~name: String~, ~description: String~,
  ~argument_hint: Option<String>~, ~disable_model_invocation: bool~,
  ~user_invocable: bool~, ~body: String~, ~base_dir: PathBuf~.
  (~base_dir~ is the parent directory of the ~SKILL.md~ file; used to resolve
  relative paths to supporting files and to allowlist reads.)
- ~parse_skill_md(source: &str, fallback_name: &str, base_dir: PathBuf) -> Result<SkillDef>~:
  - If content starts with ~---~, split at the closing ~---~ to extract YAML
    and body; otherwise treat entire content as body with empty frontmatter.
  - Deserialise the YAML block with ~serde_yaml~ (add dependency to
    ~kodo-ui/Cargo.toml~).
  - ~name~: use frontmatter value if present, else ~fallback_name~.
  - ~description~: use frontmatter value if present, else first non-empty
    paragraph of body (up to 250 chars).
  - Boolean fields default as shown in the table above.
  - Store ~base_dir~ verbatim on the struct.

** DONE Step 13: Argument substitution

*What*: Before injecting skill body into the conversation, replace placeholder
tokens with actual argument values.

*File*: ~crates/kodo-ui/src/skills.rs~ (append)

- ~render_body(body: &str, raw_args: &str) -> String~:
  - Tokenise ~raw_args~ with shell-style quoting (respect ~"…"~ and ~'…'~
    as single tokens) into ~Vec<String>~.
  - Replace ~$ARGUMENTS~ with ~raw_args~ verbatim.
  - Replace ~$ARGUMENTS[N]~ with token at index ~N~ (empty string if out of
    range).
  - Replace ~$N~ (bare digit, e.g. ~$0~, ~$1~) with token at that index.
  - If ~$ARGUMENTS~ does not appear in body at all and ~raw_args~ is
    non-empty, append ~\nARGUMENTS: <raw_args>~ to the result.
